## FINDINGS

### MEDIUM SEVERITY

#### 1. Race Condition in Thread Pool

| Attribute | Details |
|-----------|---------|
| **File** | `src/mcommon/threadpool.h` |
| **Lines** | ~100-150 |
| **Severity** | Medium |
| **Category** | Thread Safety |

**Description**: The custom `ThreadPool` implementation uses a simple queue without locks in some paths. Multiple threads accessing buffer/indexer simultaneously could cause race conditions.

**Code**:
```cpp
// threadpool.h - potential race without proper locking
m_tasks.push(task);  // not always protected
```

**Potential Exploit**: Data corruption when Zep is used with thread-enabled mode (default). The README notes threading issues in tests.

**Recommended Fix**: Add mutex protection around task queue operations, use atomic operations for counters.

---

#### 2. Command Injection in REPL

| Attribute | Details |
|-----------|---------|
| **File** | `src/mode_repl.cpp` |
| **Lines** | ~100-130 |
| **Severity** | Medium |
| **Category** | Injection |

**Description**: The REPL mode executes strings as code via `janet_run()` without sanitization.

**Code**:
```cpp
// mode_repl.cpp
auto ret = janet_run(str, "main", &out);  // user input executed
```

**Note**: This is INTENDED behavior for a REPL. However, if combined with file I/O, malicious files could execute code.

**Potential Exploit**: Opening a malicious file with REPL active could execute arbitrary code.

**Recommended Fix**: Add sandboxing, sandboxed execution mode, or confirmation before executing file content.
